## v0.8-migration-tooling

Implemented ROADMAP Phase 2.2 (v0.8 migration tooling), the last v1.0 blocker: new `ai-memory migrate` splits legacy hand-written flat sections into granular memory-store entries (verbatim heuristic chunks, LLM naming-only assist with fallback, snapshot-first, dry-run, resumable re-runs), `ai-memory init` now pre-scaffolds the 7 canonical store categories, and CHANGELOG.md was created with the release history and the store-first migration guide. Dogfooded the migration on this repo's own .ai-memory (4 legacy sections -> 15 store entries): eval went 85 -> 96/100 with zero failing checks after fixing two pre-existing eval bugs the run surfaced (store files scored for `section` instead of `store_path`; consolidated_memory.md scored via a drifted ignore-rule copy). Also fixed a timezone bug in test_hooks_e2e that failed nightly 20:00-24:00 UTC-4. Tests 264 -> 288, coverage 85.9%, ruff/mypy clean. Working tree left uncommitted for review.

**Key decisions:**
- Heuristic split always owns content (verbatim chunks); LLM only proposes names — losslessness by construction, not by prompt
- migrate bypasses regenerate_maps' fold when flipping sections to generated maps (fold would duplicate the granularized body)
- migrate is CLI-only like install (one-shot, human-supervised; keeps oversized server.py untouched); MigrateResult contract is total — no NotRequired
- Store-category scaffold derived from SECTION_TEMPLATES' generated_from so the two cannot drift
- eval identifies store files by store_path (doctor precedent); eval's ignore rule now delegates to storage's
